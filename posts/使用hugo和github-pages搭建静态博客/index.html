使用hugo和github Pages搭建静态博客

 2024.3.29  2024.3.29  3326  7 

概要

  1. 不要追寻用博客系统的方式，markdown就是一种很好的本地编辑的方式，轻量又解耦。

  2. github上创建两个repository，一个是最终呈现的静态博客，一个是用来进行编写维护的repository。

    • 比如最终呈现的是abc.github.io
    • 用来编写维护的是abc.github.io.source

  3. 两个repostory需要进行授权认证才能实现一个仓库git push之后触发另一个进行自动部署。

    没必要ssh-keygen之类的，但凡用过github都会有ssh key（公钥私钥）

    • github用户级别里面放着公钥

    • github repository的setting里面要记录着私钥

  4. 创建文章hugo new content/posts/"文章名称.md"

  5. 提交更新git push <USER_NAME>.github.io.source

  6. 目前图片有点麻烦

    • 需要在static/images中创建和文章对应的文件夹
    • 在文件夹中保存一张张图片
    • 编辑md的时候需要写相对路径，比如/文章对应的文件夹/a.png
    • 本地模式下预览有一些问题，发布之后没有问题

废话的部分

作为一个已经入行了多年的（老）技术人，维护一个看得过去的个人博客是很有必要的。

刚开始的时候还搞得很复杂，有用户系统的动态博客，需要后端数据库和nginx webserver的那种，即便加上docker的辅助也很麻烦，后来就维护不下去了，除开主观上的懒，动态博客主要是因为：

  • 需要vps或者ecs，当然现在可以serverless，不管怎么说都是一笔持续的经济成本。国内的主机往往是第一年割肉第二年宰猪，国外的主机访问延迟又很高。
  • 数据库mongodb托管的费用，这个是没有办法自己维护的。当然如果是个人博客的话，主要是自己的内容，哪里有那么多的用户数据需要保存呢。
  • 国内主机更麻烦的是还需要定期备案。

既然是个人博客，也不面向盈利，对自己而言最核心的东西就是实践费曼学习法，所以决定找个简单、省事（最后发现并没有）且不花钱的路子把博客再搞起来，一番研究后，选择了生成静态站点发布到 GitHub Pages 的方案。

工作流

整个方案的流程大致如下：

  1. 用 Markdown 格式写作文章。
  2. 使用生成器将 markdown 文件转换成静态站点。
  3. 将生成的站点内容推送到 GitHub 并发布。

写 markdown 没啥好说的，什么编辑器都可以，就Typora吧。

静态站点生成器我选择了 Hugo，此外还有 Gatsby、Jekyll、Hexo等很多选项。

接下来要做的工作是生成静态站点并通过 GitHub Pages 发布。

静态站点

使用 Hugo 生成静态博客站点非常简单，具体的步骤和用法可以参考官方文档的 Quick Start。下面简单介绍下整个过程：

  1. 安装 Hugo。macOS 下可以直接使用 homebrew安装：brew install hugo。

  2. 创建一个新的站点。这会生成一个特定目录结构的项目文件夹，用来维护所有的站点内容。假设我们想把它命名为 abc.github.io，则使用以下命令创建并切换到该目录，后续的操作和命令都会在这个根目录下执行：

    hugo new site abc.github.io
cd abc.github.io

  3. 安装一个主题。这一步是必需的，否则会因为缺少基础模板无法生成站点。安装主题有 3 种方式，以 hugo-theme-meme 主题为例：

    • 直接下载主题的压缩文件，将解压得到的文件夹重命名为主题名称 hugo-theme-meme 放到 themes/目录下。

    • 通过 git submodule 安装：

      git init
git submodule add https://github.com/reuixiy/hugo-theme-meme.git themes/hugo-theme-meme

    • 通过 Hugo Modules 安装（这种方式要求本机安装有 Go 1.12 及以上版本，且只有部分主题支持）：

      hugo mod init <abc.github.io>

      abc.github.io 并不重要，随便起个名字就行。

    安装后需要启用主题，方法是将主题名称写入到根目录下的默认配置文件 config.yml 中：

    echo 'theme = "hugo-theme-meme"' >> config.toml

  4. 添加一篇文章。可以直接在 content/posts 目录下创建 markdown 文件，但需要手动写入一些元信息，因此推荐使用 Hugo 自带的命令：hugo new content/posts/article.md，添加的文件会以如下元信息开头：

    ---
title: "article"
date: 2024-03-26T08:47:11+01:00
draft: true
---

    在下方接着写入文章内容即可。注意此时该文件为草稿状态，写作完成后需要改成 draft: false 才能部署。

  5. 启动 Hugo 预览服务器。Hugo 可以启动一个 Web 服务器，同时构建站点内容到内存中并在检测到文件更改后重新渲染，方便我们在开发环境实时预览对站点所做的更改。

    hugo server -D

    添加 -D 选项以输出草稿状态的文章，执行成功后可以通过 http://localhost:1313/ 访问站点。

  6. 自定义主题配置。站点的配置项默认保存在根目录的 config.toml 文件中，配置项较多时通常会用主题提供的预设配置文件来替换该文件，还可以通过config 目录加多个文件的方式来组织配置。默认配置文件如下：

    baseURL = "http://example.org/"      # 发布地址，由主机名以及路径组成
languageCode = "en-us"               # 语言代码，中文可以设置为"zh"
title = "My New Hugo Site"           # 站点标题

    这一步应该是整个过程中最麻烦也是最容易出问题的一步，视乎你选择的主题与想要的功能不同，需要自定义的配置项也不同，数量从几个到上百个不等。有些主题会有详细的文档解释配置过程，有些则一笔带过只能自己去摸索，配置较多时相互间可能还有依赖关系，最好更改一个配置就刷新一次页面确认下结果。

    建议起步时一切从简，花大把时间搞各种花里胡哨的样式和功能，还不如多写几篇文章。

  7. 构建静态页面。站点配置成我们理想的效果之后就可以构建静态页面了：

    hugo -D

    添加 -D 选项可以在结果中包括草稿内容，默认情况下静态页面会输出到根目录下的 public 文件夹中。

通过github pages发布

这一步 Hugo 的官方文档同样在 Host on GitHub 中进行了详细的介绍，并且还很贴心的提供了自动化操作的 Shell 脚本。

有两种方式：

  1. 通过个人主页发布：必须创建一个 abc.github.io 仓库来托管生成的静态内容，发布后的域名为 https://abc.github.io。
  2. 通过项目主页发布：可以随意创建 <PROJECT_NAME> 仓库，发布后的域名为 https://.github.io/<PROJECT_NAME>。

发布步骤

  1. 在 GitHub 创建个人主页仓库，仓库名称必须设置为 <PROJECT_NAME>.github.io，这个仓库仅存放生成的静态内容。

    比如我的最终的仓库如下

  2. 在 GitHub 创建一个项目仓库 hugo-blog 并添加为我们本地项目文件夹的远程仓库。这个仓库用来维护站点配置和原始的文章内容。

    比如我的中间维护仓库如下

  3. 假设我们在已经通过上文的步骤在 public 文件夹中生成了想发布的静态内容，运行：

    git submodule add -b main https://github.com/<USERNAME>/<USERNAME>.github.io.git public

    在 public 目录中创建一个 git 子模块，之后这个目录将以 https://github.com//<USERNAME>.github.io 作为远程仓库。

  4. 确保配置文件中的 baseUrl 已经设置为了 .github.io。

  5. 针对中间维护的仓库，也就是tigbox.github.io.source仓库进行git push。

    大功告成！稍等几分钟就可以在 https://.github.io 看到我们的个人博客了。

通过github actions自动部署

「创作-发布」流程如下：

  1. 在项目仓库编辑原始内容并进行版本管理。
  2. 执行自动脚本生成静态站点并推送到个人主页仓库完成发布。

这套流程已经很流畅，但还有一些改进空间：我们可以使用 GitHub Actions，在每次向远程的项目仓库推送原始内容时自动触发第 2 步进行部署。

GitHub 上有许多这类自动化部署任务的开源 Actions 项目，我们选择了其中一个简单易用的 GitHub Actions for Hugo。具体的操作步骤截图和详细配置项可以查看该项目的 README。下面简单介绍下配置过程：

on:

    runs-on: ubuntu-20.04

      - name: Setup Hugo

        with:
          hugo-version: '0.108.0'

      - name: Update theme
        run: git submodule update --init --recursive

      - name: Build
        run: hugo --gc --verbose --minify

      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3

          deploy_key: ${{ secrets.ACTIONS_DEPLOY_KEY }} # 这里的 ACTIONS_DEPLOY_KEY 则是上面设置 Private Key的变量名

          publish_dir: ./public/
          keep_files: false # remove existing files
          publish_branch: main  # deploying branch
          commit_message: ${{ github.event.head_commit.message }}

    这个文件所定义的 workflow 基于项目仓库运行，但我们需要将运行过程生成的静态文件推送到个人主页仓库 <USER_NAME>.github.io 完成发布，因此在 Deploy 任务中按照文档的 Deploy to external repository external_repository 一节做了专门的配置。

    总结

    1. 个人博客是费曼学习法很好的一种实践方式，没必要用数据库加webserver，静态博客足矣。

    2. 用markdown来编辑内容，门槛不高。那些博客系统就显得太重量级了。

    3. 需要有两个repository，一个是最终的静态文件仓库，一个是用来进行维护控制的中间仓库。

      两个仓库需要进行ssh的认证，来达自动触发部署。
